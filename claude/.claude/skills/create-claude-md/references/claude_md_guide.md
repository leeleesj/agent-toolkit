# CLAUDE.md 작성 가이드 — 파워 유저 실전편

> Claude Code의 CLAUDE.md를 효과적으로 작성하고 관리하기 위한 실전 가이드.
> 공식 문서, 파워 유저 사례, Anthropic 팀 내부 워크플로우를 종합 정리.

---

## 1. CLAUDE.md란?

CLAUDE.md는 Claude Code가 **매 세션 시작 시 자동으로 읽는 유일한 파일**이다.
Claude는 세션 간 기억이 없기 때문에, 프로젝트에 대해 알아야 할 모든 필수 맥락을 이 파일에서 얻는다.

핵심 비유: **신규 입사자에게 주는 온보딩 문서**가 아니라, **시니어 개발자가 옆에 앉았을 때 꼭 알려줘야 할 것만 적은 메모**에 가까워야 한다.

## 2. 핵심 원칙

### 원칙 1: "짧게, 핵심만"

| 기준 | 권장 |
|---|---|
| 분량 | **200줄 이하**, 가능하면 50~100줄 |
| 토큰 | 약 500~1500 토큰 |
| 비유 | 신입 온보딩 문서(X) → 시니어에게 건네는 메모(O) |

왜 중요한가:
- CLAUDE.md는 **매 메시지마다** 컨텍스트에 포함된다
- 400줄짜리 CLAUDE.md = 코드 한 줄 고칠 때도 5000토큰 소모
- 프론티어 모델도 ~150~200개의 지시만 안정적으로 따를 수 있다
- 문서가 길수록 중요한 규칙이 다른 내용 사이에 묻힌다

### 원칙 2: "Claude가 실수할 때만 추가한다"

Claude Code를 만든 Boris Cherny의 접근법:

> "Claude가 뭔가를 잘못할 때마다 CLAUDE.md에 추가해서,
> 다음 세션에서는 같은 실수를 안 하게 한다."

이것이 가장 효과적인 CLAUDE.md 성장 방식이다:
1. 처음엔 최소한의 내용으로 시작
2. 작업 중 Claude가 실수하면 → 해당 규칙을 한 줄 추가
3. 세션 끝날 때 정리
4. 반복

### 원칙 3: "린터가 할 수 있는 일은 넣지 않는다"

LLM에게 코드 스타일을 강제하는 건 비효율적이다.
- 코드 포맷팅 → hooks에서 `black`, `prettier` 실행
- import 정렬 → hooks에서 `isort` 실행
- 타입 체크 → LSP 플러그인 사용

CLAUDE.md에는 **"왜"에 해당하는 설계 결정**만 남긴다.

### 원칙 4: "Claude가 파일을 읽어서 알 수 있는 건 빼라"

| 정보 | CLAUDE.md에 넣기 | 이유 |
|---|---|---|
| `.env`의 변수 목록 | ❌ | Claude가 `.env` 직접 읽을 수 있음 |
| `pyproject.toml` 의존성 | ❌ | Claude가 직접 읽을 수 있음 |
| DB 스키마 전체 | ❌ | Claude가 DB 파일이나 모델 코드 읽으면 됨 |
| date 컬럼이 ISO vs 표시용으로 분리된 이유 | ✅ | 코드만 봐서는 의도를 모름 |
| 두 RAG 컬렉션이 같은 임베딩 모델을 써야 하는 이유 | ✅ | 설계 의도를 모르면 실수함 |

## 3. 넣어야 할 것 vs 빼야 할 것

### ✅ 넣어야 할 것

```
1. 프로젝트 한 줄 요약
   → Claude가 "이게 뭐 하는 프로젝트인지" 즉시 파악

2. 기술 스택 (한 줄)
   → 프레임워크, DB, 주요 라이브러리만

3. 핵심 경로 (5~10개)
   → 가장 자주 건드리는 파일/디렉토리만

4. 설계 규칙 (Claude가 어기면 버그 나는 것들)
   → "반드시", "절대" 같은 표현으로 강조
   → 예: "두 컬렉션은 반드시 같은 임베딩 모델 사용"

5. 코드 컨벤션 (도구로 강제할 수 없는 것만)
   → 커밋 메시지 형식, 언어 설정 등

6. 현재 알려진 약점/주의사항
   → Claude가 특히 조심해야 할 맥락

7. 참조 문서 경로
   → 상세 내용은 별도 파일, 여기선 경로만
```

### ❌ 빼야 할 것

```
1. TODO / 로드맵 / 다음 할 것
   → docs/ROADMAP.md로 분리, 필요시 @로 참조

2. 포트폴리오 스토리라인 / 마일스톤
   → 사람용 문서, Claude한테 불필요

3. 서버 실행 방법 / 인프라 설정
   → docs/SETUP.md로 분리

4. 환경변수 상세 목록
   → .env 파일에서 직접 읽으면 됨

5. DB 스키마 전체 덤프
   → 설계 의도만 남기고, 스키마는 코드에서 읽기

6. 패키지 의존성 목록
   → pyproject.toml에서 읽으면 됨

7. 구현 가이드 / 코드 예시
   → Skills로 분리

8. 평가 결과 상세 수치
   → 결론만 남기고, 상세는 별도 파일 참조

9. 코드 스타일 규칙
   → hooks나 린터로 강제
```

## 4. CLAUDE.md 판단 체크리스트

특정 내용을 CLAUDE.md에 넣을지 고민될 때, 아래 세 가지를 순서대로 확인:

```
Q1. Claude가 이걸 모르면 실수하는가?
    → No → 빼기
    → Yes → Q2로

Q2. 매 세션마다 필요한가?
    → No → 별도 파일로 분리, 필요시 @로 참조
    → Yes → Q3로

Q3. Claude가 다른 파일을 읽어서 알 수 있는가?
    (.env, pyproject.toml, 코드 자체 등)
    → Yes → 빼기
    → No → CLAUDE.md에 넣기 ✅
```

## 5. CLAUDE.md 템플릿

```markdown
# CLAUDE.md — [프로젝트명]

## 프로젝트
[한두 문장으로 뭘 하는 프로젝트인지]

## 기술 스택
[프레임워크 + DB + 핵심 라이브러리, 한 줄]

## 핵심 경로
- path/to/main.py — [역할]
- path/to/core/ — [역할]
- path/to/tests/ — [역할]

## 설계 규칙
- [Claude가 어기면 버그 나는 규칙 1]
- [Claude가 어기면 버그 나는 규칙 2]
- [설계 의도를 모르면 실수하는 규칙 3]

## 현재 상태 / 약점
- [지금 알려진 문제점, 개선 대상]

## 코드 규칙
- [커밋 형식, 언어 설정 등 도구로 강제 못 하는 것]

## 참조 문서
- 상세 로드맵: docs/ROADMAP.md
- 설정 가이드: docs/SETUP.md
- 평가 결과: evaluation/baseline_results.md
```

## 6. CLAUDE.md 위치와 스코프

CLAUDE.md는 네 곳에 둘 수 있다:

| 위치 | 적용 범위 | 용도 |
|---|---|---|
| `./CLAUDE.md` (프로젝트 루트) | 이 프로젝트만 | 팀 공유, git 커밋 |
| `./CLAUDE.local.md` | 이 프로젝트, 나만 | 개인 설정, .gitignore 자동 추가 |
| `./.claude/CLAUDE.md` | 이 프로젝트만 | 팀 공유, 설정 파일과 함께 관리 |
| `~/.claude/CLAUDE.md` | 모든 프로젝트 | 글로벌 개인 습관 |

일반적인 조합:
- **프로젝트 CLAUDE.md** → 프로젝트 규칙 (팀원도 사용)
- **CLAUDE.local.md** → 나만의 프로젝트별 설정 (git에 올라가지 않음)
- **글로벌 CLAUDE.md** → 개인 선호 (한국어 답변, 커밋 스타일 등)

### CLAUDE.md 빠르게 만드는 법
- **직접 작성**: 템플릿 기반으로 작성 (섹션 5 참조)
- **자동 생성**: 프로젝트 폴더에서 `/init` 실행
  → Claude Code가 코드베이스를 분석해서 초안 자동 생성
  → 초안을 검토하고 불필요한 내용 제거 (섹션 3 기준으로)

### 다른 파일 참조 (@import)
CLAUDE.md 안에서 `@파일경로`로 다른 파일을 참조할 수 있다:

```markdown
## 참조 문서
프로젝트 개요는 @README.md 참고
아키텍처 상세는 @docs/architecture.md 참고
```

→ CLAUDE.md를 간결하게 유지하면서 상세 내용은 분리 가능
→ Claude가 필요할 때만 참조 파일을 읽으므로 토큰 절약

## 7. CLAUDE.md 수정 후 인식 방식

CLAUDE.md는 **세션 시작 시 한 번 로드**된다.

| 수정 상황 | Claude 인식 여부 | 대처법 |
|---|---|---|
| Claude Code가 직접 수정 | ✅ 자동 인식 | 별도 조치 불필요 |
| 내가 에디터에서 수정 (같은 세션) | ❌ 자동 인식 안 됨 | `@CLAUDE.md` 로 다시 불러주기 |
| 수정 후 새 세션 시작 | ✅ 자동 인식 | 별도 조치 불필요 |

권장 패턴:
- 세션 끝날 때 CLAUDE.md 수정
- 다음 세션에서 자연스럽게 반영
- 급하면 `@CLAUDE.md 업데이트된 내용 확인해줘`

## 8. 성장시키는 방법 — "실수 기반 점진적 추가"

### 단계 1: 최소한으로 시작
처음엔 10줄짜리 CLAUDE.md로 시작한다.
프로젝트명, 기술 스택, 핵심 규칙 3개면 충분하다.

### 단계 2: 실수를 기록
작업 중 Claude가 잘못한 것을 메모한다.
- "date 컬럼에 한국어 형식 문자열을 넣었다" → 규칙 추가
- "ChromaDB 컬렉션마다 다른 임베딩 모델을 썼다" → 규칙 추가
- "커밋 메시지에 'AI-generated'를 넣었다" → 규칙 추가

### 단계 3: 세션 끝에 정리
메모한 실수를 CLAUDE.md에 한 줄씩 추가한다.
설명이 아니라 **규칙**으로 적는다.

나쁜 예:
```
date 컬럼은 ISO 형식이고 이건 시계열 분석을 위해서이며,
date_display는 한국어 표시용으로 분리했는데 일부 없을 수도 있다.
```

좋은 예:
```
- date 컬럼: ISO 형식만 사용. date_display: 한국어 표시용. 혼용 금지.
```

### 단계 4: 주기적 정리
한 달에 한 번, CLAUDE.md를 리뷰한다:
- 더 이상 해당 안 되는 규칙 삭제
- 여러 줄로 흩어진 관련 규칙 한 줄로 병합
- 200줄 넘으면 반드시 정리

## 9. CLAUDE.md vs Skills vs Hooks 역할 분담

| 내용 | 어디에 |
|---|---|
| "두 컬렉션은 같은 임베딩 모델 사용" | **CLAUDE.md** (설계 규칙) |
| "평가 파이프라인 실행 절차" | **Skill** (반복 워크플로우) |
| "코드 수정 후 자동 black + isort" | **Hook** (결정론적 강제) |
| "MTEB 평가 + 결과 비교 분석" | **Skill** (전문 지식) |
| "커밋 메시지에 AI 단어 포함 시 차단" | **Hook** (100% 강제) |
| "한국어로 답변" | **CLAUDE.md** (글로벌 규칙) |

판단 기준:
- **매번 알아야 하는 설계 의도** → CLAUDE.md
- **반복 수행하는 작업 절차** → Skill
- **100% 자동 실행해야 하는 검증** → Hook

## 10. 자주 하는 실수

### 실수 1: "설계 문서"를 CLAUDE.md로 쓰기
CLAUDE.md는 설계 문서가 아니다. 설계 문서는 `docs/`에 두고 필요시 참조한다.

### 실수 2: 처음부터 완벽하게 만들려고 하기
10줄로 시작해서 실수가 생길 때마다 한 줄씩 추가하는 게 최선이다.

### 실수 3: TODO를 CLAUDE.md에 넣기
Claude는 세션 간 기억이 없다. TODO는 별도 파일에 두고 필요한 세션에서만 불러온다.

### 실수 4: 코드 예시를 CLAUDE.md에 넣기
코드 가이드는 Skill로 분리한다. CLAUDE.md에는 규칙만 남긴다.

### 실수 5: 200줄을 넘기기
넘기면 무조건 리팩토링한다. 예외 없다.
